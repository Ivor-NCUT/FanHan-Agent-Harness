# 内容风格守门

## 目标

把用户给出的中文内容规则变成可执行的写作约束，减少模板感、空话、硬连接和常见 AI 腔。

这个 skill 负责“内容交付前的风格把关”。它不替代具体写作类 skill；当其他写作 skill 已经产出草稿时，用本 skill 做最后一轮检查和修正。

## 触发场景

用户的任务属于这些类型时使用：

- 写文章、公众号、小红书、短视频文案、口播稿、演讲稿、课程稿、宣传文案。
- 润色、改写、降 AI 味、去模板感、改得更自然。
- 标题、开头、结尾、脚本、内容诊断。
- 需要对外发布或给真实读者阅读的中文文本。

这些任务不默认使用：

- 普通代码修改、技术排查、命令执行、数据清洗。
- 内部过程说明、工具日志、测试报告。
- 用户只要事实检索或文件处理结果，不要求文案表达。

如果一个技术任务最终需要写 README、发布说明、用户邮件或对外介绍，只对那段面向读者的文字使用本 skill。

## 工作方式

### 1. 判断文本用途

先判断文本要给谁看、在哪里发布、读者读完要做什么。若用户已经说明平台和对象，直接使用。若缺失但不影响处理，按最接近的使用场景推断，不反复追问。

### 2. 写作或改写

保持用户原本的事实、观点、案例和立场。可以调整顺序、句子长度、节奏和表达密度，但不要替用户编造经历、数据、身份、客户案例或外部评价。

优先使用这些表达方式：

- 具体动作、具体场景、具体对象。
- 短句和自然口语，但不要故意粗糙。
- 让判断落在可感知的细节上。
- 少用概念堆叠，多写“发生了什么、谁在做、哪里卡住、怎么变化”。

### 3. 禁用句式扫描

交付前检查并改掉这些句式，除非用户要求直接引用原文。

**对比强调**

- `不是……而是……`
- `……不是关键，……才是`

**背景引入**

- `在……的浪潮下`
- `随着……的发展`

**重要性强调**

- `……是重中之重`
- `……是关键所在`
- `……是……的核心`

**手段与目的**

- `通过……，实现……`
- `以……为抓手`
- `……赋能……`

**总结概括**

- `首先，其次，最后`
- `综上所述`
- `总而言之`

不要只做文字替换。要根据上下文重写句子，让表达自然成立。

### 4. 语气和结构扫描

检查这些问题：

- 过度正式、书面化，例如“在当前的社会环境下”。
- 缺乏具体细节，例如“取得了巨大成功”。
- 模板化结构，例如总分总、背景问题方案。
- 生硬连接词，例如“首先”“其次”“然后”“因此”“然而”。
- 同一关键词在 100 字内出现 2 次或更多。
- 专业术语堆在一起但没有解释。
- 英文缩写首次出现没有中文解释，例如 AI、SaaS。

遇到关键词重复时，优先用更具体的名词、动作或上下文省略解决，不要机械替换成近义词。

### 5. 例外处理

这些情况允许保留原词：

- 直接引用他人原文，引用处加引号并注明来源或说明为用户原文。
- 用户明确要求使用某个禁用词或固定表达。
- 专有名词、产品名、组织名不能改。

即使有例外，也不要连续两句以上使用禁用句式或禁用词。

## 输出要求

用户要成稿时，直接给修改后的成稿。不要在成稿前解释一大段规则。

用户要诊断时，按这个格式：

```markdown
**主要问题**
- [具体问题]：[原句或位置] → [为什么影响阅读]

**修改方向**
- [可执行建议]

**改写版本**
[完整改写文本]
```

用户只要“帮我改自然一点”时，输出改写版本即可；如有必要，在末尾用 1-3 条说明关键改动。

## 自检清单

交付前快速检查：

- 是否保留了用户原事实和立场。
- 是否移除了禁用句式。
- 是否减少了空话、套话和硬连接。
- 是否避免同一关键词在短距离内重复。
- 英文缩写首次出现时是否有中文解释。
- 成稿是否像真人写给具体读者看的内容。
