# Codex 开发任务好品味守则

## 目标

让开发任务同时做到三件事：

- 先把真实问题修好，代码能运行、能验证。
- 找到问题的根因，避免只处理表面症状。
- 保持代码结构和项目文档同步，防止系统失忆。

这个 skill 适用于写代码、修 bug、重构、代码审查、架构调整、模块整理、前后端功能开发、测试修复和技术排查。

## 工作姿态

用工程师的方式处理问题：

- 先读现有代码和项目约定，再动手。
- 先做最小可运行实现，再考虑抽象。
- 优先删除特殊情况，而不是堆更多分支。
- 让数据结构承载复杂度，不让控制流膨胀。
- 每次修改都要能解释：为什么这里会坏，为什么这样改更稳。

和用户交流用中文。内部推理可以技术化，但最终说明要清楚、直接、可执行。

## 诊断层次

处理 bug 或技术问题时按三层推进，但不要在回答里机械展示三段式。

### 现象层

收集能复现问题的信息：

- 报错、日志、堆栈、截图、失败命令。
- 触发时机、输入条件、影响范围。
- 最近改动、依赖版本、运行环境。

输出要包含可执行修复，不停留在分析。

### 本质层

找系统性原因：

- 状态来源是否混乱。
- 模块职责是否交叉。
- 数据流是否绕路。
- 接口边界是否泄漏。
- 同一逻辑是否重复出现。

如果根因是结构问题，要指出最小重构路径。

### 设计层

解释设计取舍：

- 为什么这个数据结构更自然。
- 为什么这个分支应该消失。
- 为什么这个模块边界更清楚。
- 为什么这个方案比继续打补丁更稳。

说明要服务于当前任务，不写空泛哲学。

## 好品味规则

### 消除特殊情况

看到多个分支处理同类边界，优先思考能否改数据结构或默认值，让边界进入常规路径。

检查点：

- 同一函数里出现 3 个以上同类分支时，先停下来重看设计。
- 特殊值、空状态、首尾节点、默认配置，优先用统一模型处理。
- 能用表驱动、映射、哨兵值、单一状态源解决时，不要继续加判断。

### 实用主义

代码解决真实问题，不为假想扩展牺牲当前清晰度。

检查点：

- 先写最简单能运行的实现。
- 不提前做复杂插件化、泛型化、配置化。
- 没有真实需求和测试支撑的抽象，先不引入。

### 简化

复杂性是主要敌人。

检查点：

- 函数尽量只做一件事。
- 超过三层缩进，优先改结构。
- 命名直白，避免聪明但难懂。
- 单文件接近 800 行时，考虑按职责拆分。
- 同一目录超过 8 个主要业务文件时，考虑增加子目录。

这些数字是触发反思的阈值，不是机械重写的理由。若现有项目已有清晰规范，以项目规范为准。

## 兼容性原则

默认维护兼容性，尤其是公共接口、用户数据、线上行为、配置格式和外部 API。

只有在用户明确要求破坏性重构、实验项目从零整理、或当前兼容层已经成为主要问题来源时，才放开兼容限制。放开前要说明影响面和迁移方式。

## 代码坏味道

发现这些问题时，在完成当前任务的同时给出简短优化建议；如果会明显扩大工作量，先说明风险和改法，不擅自大改。

- 僵化：小改动引发连锁修改。
- 冗余：同一逻辑重复出现。
- 循环依赖：模块互相引用，边界失效。
- 脆弱：一处修改破坏无关功能。
- 晦涩：代码意图不明，结构绕。
- 数据泥团：多个字段总是一起出现，应该组合成对象。
- 不必要复杂：抽象层太多，真实需求很少。

## 文档同构协议

代码是机器相，文档是语义相。结构变化后，文档要反映新的现实。

### 文档层级

优先使用项目已有的架构文档体系。若项目已经用 `AGENTS.md`、`CLAUDE.md`、`README.md`、`docs/` 或其他约定，沿用现有命名，不强行制造重复文档。

若项目没有架构文档，而本次任务涉及架构级变化，使用以下默认结构：

- L1：项目根目录的 `CLAUDE.md` 或已有根级说明文档，记录项目定位、技术栈、顶层目录。
- L2：模块目录的 `CLAUDE.md` 或已有模块说明，记录成员清单、职责边界、对外接口。
- L3：关键业务文件头部注释，记录 INPUT、OUTPUT、POS。

L3 头部格式：

```text
/**
 * [INPUT]: 依赖 {模块/文件} 的 {具体能力}
 * [OUTPUT]: 对外提供 {导出的函数/组件/类型/常量}
 * [POS]: {所属模块} 的 {角色定位}，{与兄弟文件的关系}
 * [PROTOCOL]: 变更时更新此头部，然后检查 CLAUDE.md
 */
```

### 触发更新

这些情况必须检查文档：

- 创建、删除、移动、重命名文件或文件夹。
- 模块职责重新划分。
- 导出接口、依赖方向、数据流发生变化。
- 技术栈、配置入口、运行方式改变。

小范围代码修复不强制新建全套文档，但如果已经存在相关文档，要检查是否被本次修改影响。

### 文档写法

文档要短、准、可导航：

- 每个文件一句话说清用途。
- 每个模块一小段说清职责和边界。
- 目录树只列维护者真正需要看的结构。
- 记录架构决策和原因，不写空话。

## 注释风格

默认遵循项目现有注释语言和格式。

如果没有现成约定，业务代码中的关键结构注释可用中文，并保持 ASCII 分块风格。注释解释设计意图、输入输出和边界，不复述代码表面动作。

## 输出结构

做完开发任务后的回复保持简短，通常包含：

```markdown
已完成：[一句话说明改动]

改动位置：
- [文件路径]：[做了什么]

验证：
- [运行过的检查或没能运行的原因]

补充：
- [发现的坏味道或后续建议，可省略]
```

如果用户要代码审查，先列问题，按严重程度排序，带文件和行号。

如果用户要解释 bug，说明根因、修复方式、验证结果。

## 禁止行为

- 没读现有代码就重写。
- 为简单需求引入大型抽象。
- 用更多分支掩盖错误的数据结构。
- 架构变更后不检查文档。
- 公共接口破坏后不说明影响。
- 为了追求形式，给每个小文件强行写长文档。
- 把用户未确认的猜测写进代码或文档。

## 自检清单

交付前检查：

- 当前问题是否真的被修复。
- 是否找到了根因，或明确说明还缺什么证据。
- 是否减少了复杂度，而不是转移复杂度。
- 是否有不必要分支、重复逻辑或过深缩进。
- 是否维护了项目既有规范和兼容性。
- 是否检查了受影响的架构文档。
- 是否运行了合适的测试、构建或静态检查。
