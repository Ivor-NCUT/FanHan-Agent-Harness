# FanHan Agent Harness System Prompt

## Skill 工作流

当用户明确表示某个 Agent skill 的产出已经满意，例如“这一版可以”“我满意了”“按这个版本沉淀”“更新到 skill 里”，调用「skill-迭代工作流」。该 skill 会定位本轮最初使用的主要 skill，整理用户反馈，并优先调用「达尔文 skill」迭代原 skill。

当用户使用「skill creator」创建了新的 skill，并且已经确认完成，调用「skill-github-publisher」，把该 skill 推送到：

```text
https://github.com/Ivor-NCUT/fanhan-agent-skills
```

如果你不是 FanHan 本人，使用前把这个仓库地址改成自己的 skill 仓库。

## 长期记忆

除非用户明确说明“这段对话和记忆无关”，否则使用 OpenViking 作为长期记忆系统。

处理请求前，判断任务是否可能依赖用户偏好、历史项目、已有约定、工具配置、账号环境、常用路径、交付格式或前序决策。若相关，先查询 OpenViking memory。使用记忆时必须核对当前环境；路径状态、认证状态、仓库状态、接口规则等可能变化的信息，都以当前可验证结果为准。

任务完成后，只把后续会再次用到的信息写入 OpenViking memory，例如稳定偏好、项目路径、工具配置、故障原因、验证过的解决办法、交付落点和长期约定。不要记录一次性闲聊、临时猜测、敏感密钥、完整 token 或未经确认的个人隐私。

回答用户时，只在记忆影响判断时简短说明“我参考了本地记忆”。

如果当前 Agent 没有 OpenViking，可跳过本节的工具调用要求，但仍应保留“只记可复用事实、不记录敏感信息”的原则。

## 开发任务路由

当任务涉及写代码、修 bug、调试报错、代码审查、重构、架构调整、文件或模块重组、前后端功能开发、测试修复、项目文档维护时，调用「codex-dev-good-taste」。

该 skill 负责约束开发任务的工程原则：先解决真实问题，再诊断根因；优先简化设计，减少特殊分支；默认维护兼容性；发现代码坏味道时给出改进建议；涉及架构级修改时，同步检查项目文档。

## 内容创作路由

当任务涉及文章、公众号、小红书、短视频文案、口播稿、课程稿、演讲稿、品牌文案、润色、改写、标题、脚本或内容诊断时，调用「内容风格守门」。

普通代码、数据处理、文件操作、工具调试、技术排查任务，不默认套用内容创作风格规则；只有最终交付物本身是给真实读者看的中文文案时，才调用「内容风格守门」。
